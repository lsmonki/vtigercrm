1. Introduction

The vtiger CRM is a Customer Relationship Management software for your organization-wide sales, marketing, customer support, and product management requirements. Security Management is one of the primary modules in vtiger CRM that provides access level control (ACL) to various functions to the users according to their privileges in an organization.  With  User Management, the users for your organization can be created and their access privileges can be defined.

Security Management consists for the following sub-modules:

  • Users
  • Roles
  • Profiles
  • Groups
  • Default Organization Sharing Access
  • User Defined Sharing Rules
  • Default Organization Field Access

2. Users

Users in user management facilitate to Add/Edit/Delete/View Users.  There are two types of users in vtiger CRM software, Standard Users and Administrator users.

Standard Users can perform only CRUD (Create, Retrieve, Update, and Delete) operations on records such as, leads, accounts, contacts, potentials, trouble tickets, and others. Where as Administrators can manage the complete software that includes, managing users/groups & their access privileges, customizing vtiger CRM user interface, creating communications templates and configuring all organization wide settings.  In general Standard Users will not have access to the Settings Panel and only Admin Users will have access to the Settings Panel

How to create a Administrator User and Standard User?

In the user creation screen there is a check box called Admin. When a user is created with selecting the Admin option then the user is an Administrator User and if the User is created without selecting the Admin option then the user is a Standard User.

Notes

  • Each user created in the system must be associated with a ROLE
  • Admin User will have all privileges in the organization
  • Standard User's  privilege will be defined by his  Role

The other functions in User management are Change Password, Deactivate Users, List Mail Server, Home Page Order, and View Login History.

3. Roles

The organization level hierarchy can be defined using Roles. Roles control the visibility level that users have in the organization’s data. Users at any given role level can always view, edit, delete  all data owned by users below them in the hierarchy . For example, the Sales Manger can access/edit/delete all the Sales Reps' records because they are in the higher hierarchical level in the role tree. The Sales Reps can access only their records.

Roles in user management facilitate to Add/Edit/Delete/View Roles. Multiple profile support is available in roles i.e. a role can consists of one or more number of profiles.

Notes

  • Users in the higher level of hierarchy cannot view the records that are shared to their subordinate users
  • Role must be associated to at least one profile

4. Profiles

Profile function provides access to vtiger CRM modules (tabs), standard actions, utility actions , fields in various modules and global permissions. Users associated to the specific profile can access the functions that are assigned to them.

Profiles defines the access privileges for the following:

4.1. Global Privileges

Global Privileges define the global permissions for all modules and all actions. Global Privileges consists of :

  • View all: can view all  module data in the organization
  • Edit all: can create/edit all module data in the organization

4.2. Tab Privileges

Tab Privileges define the module tab level access for the following modules:

Leads, Accounts, Contacts, Potentials, Dashboards, Activities, Trouble Tickets, FAQ, Calendar, Price Books, Purchase Orders,  Invoices, Reports, Notes, Emails, Products, Vendors, Quotes, Sales Orders, RSS, Campaigns

4.3. Standard Privileges

Standard Privileges define the permissions for Create/Edit, View, Delete Actions in the following modules:

-Leads, Accounts, Contacts, Potentials, Activities, Trouble Tickets, FAQ, Price Books, Purchase Order,  Invoices, Notes, Emails, Products, Vendors, Quotes, Sales Orders

4.4. Field Privileges

Field Privileges define the access permissions for the fields (standard fields/custom fields) in the following modules:

Leads, Accounts, Contacts, Potentials, Activities, Trouble Tickets, FAQ, Price Books, Purchase Orders,  Invoices, Notes, Emails, Products, Vendors, Quotes, and Sales Orders.

You can also define Field Privileges for the custom fields.

4.5. Utility Privileges

Utility Privileges define the permissions for the actions Import, Export, Merge, Convert Lead 
       
Profiles in user management facilitate to Add/Edit/Delete/View Roles.

Notes

  • Global Privileges in Profiles override the permissions defined by Tab, Standard, Utility and Field Privileges.  For example, in a profile the access for Potentials tab is denied via Tab Privileges. Even then the Profile User can view the Potentials module data if the 'View all'  permission is allowed in the global privileges of that profile.
  • Privileges defined in the profile apply to only non-admin users. For admin users the permissions defined in profile is immaterial.
  • Profiles override the Default Organization Sharing Rules and the User defined Sharing Rules.  For example consider that the organization sharing rule allows an user to view the Potentials of other users. But the user cannot view the Potentials module because his profile does not allow him to access the Potential module.

5. Groups

Groups can be created to manage a set of common records.  A group can consist of the following:

  • Groups --> group can consists of sub groups. All the users present in this sub group will also be in the main group.
  • Roles --> all the users under the given role will be in the group
  • Roles and Subordinates --> all the users under the given role and role subordinates will be in the group.
  • User --> the specified user will be present in the group

Users associated with a particular group can access the records assigned to the group and perform the necessary CRUD operations on the records.

Groups in user management facilitate to Add/Edit/Delete/View Roles.

Notes

  • A user, role, role subordinate, group can be present in any number of groups
  • When a group is assigned as owner to a crmentity, all the members  of the group can access the entity. When the ownership is changed to any one member in the group  then only that member can access the entity and all other members in the group cannot access the entity. If in this  case it is required that all the members of the group should have access to the entity, then user defined sharing rule should be defined to share all the data between the group members

6. Default Organization Sharing Access

Default Organization Sharing Access controls the data sharing at organization level. Other users can access the owners' records as per organization-level data (records in various modules) access privilege. You can provide the following types of access levels to vtiger CRM modules:

  • Private: Only the record owner, and users above that role in the hierarchy, can view, edit, and delete on those records.
  • Public: Read Only: All users can view the records but cannot edit them. Only the owner, and users above that role in the hierarchy, can edit/delete those records.
  • Public: Read Create/Edit: All users can view/create/edit the records but cannot delete them. Only the owner, and users above that role in the hierarchy, can delete those records.
  • Public: Read Create/Edit, Delete:  This sharing privilege provides global access for all the users. All the users can view/create/edit/delete other users records in the organization.

Default Organization sharing privileges can be defined for the following modules:

Accounts & Contacts, Potentials, Leads,  Emails, HelpDesk, Quotes, Purchase Order, Sales Order, Invoice

For Activities module the default organization sharing privilege value is Private and it is a fixed value and this cannot be altered.

By default, the default organization sharing privilege value for all the modules will be Public:Read Create/Edit, Delete

Notes

If the Default Organization Sharing Privilege is set as Private for Accounts & Contacts, then the default organization sharing privilege value for the following modules will also be set as a private:

Potentials,  Emails, HelpDesk, Quotes, Purchase Order, Sales Order and Invoice

7. User Defined Sharing Access

User Defined Sharing Rules allows the administrators to selectively grant data access to a  set of Users. Data Sharing rules can be created to share data between the following modules:

  • From Role to Role
  • From Role to Role Subordinates
  • From Role to Group
  • From Role Subordinates to Role
  • From Role Subordinates to Role Subordinates
  • From Role Subordinates to Groups
  • From Group to Role
  • From Group to Role Subordinates
  • From Group to Group

Sharing Rules can be created for the following modules:

1. Leads Sharing Rules:

Leads owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

These lead related Emails will also be shared in the permission of Read Only/ Read Write.

2. Accounts Sharing Rules: 

Accounts owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

These account related Potentials, Tickets, Quotes, Sales Order, Invoice, Emails will also be shared in the permission of Read Only/ Read Write.

Note

Sharing Rules created for Accounts module will also apply to the contacts module

3. Potentials Sharing Rules:

Potentials owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

These potential related  Quotes and Sales Order will also be shared in the permission of Read Only/ Read Write.

4. HelpDesk Sharing Rules:

Tickets owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write. 
            
5. Email Sharing Rules:

Emails owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

6. Quotes Sharing Rules:

Quotes owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

These quote related  Sales Order will also be shared in the permission of Read Only/ Read Write.

7. Purchase Order Sharing Rules:

Purchase Orders owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

8. Sales Order Sharing Rules:

Sales Order owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

These sales order related  Invoice will also be shared in the permission of Read Only/ Read Write.

9. Invoice Sharing Rules:

Invoices owned by the Users of a given Role/Role Subordinates/Group can be shared with users of a  Role/Role Subordinates/Group in the permission of  Read Only/Read Write.

Notes

  • Sharing Rules can be used only to widen the data access and they cannot be used to restrict data access
  • Any number of sharing rules can be defined for a single Role, Role Subordinates and Group
  • Sharing rules are applicable to all existing data and the data that will be created in future
  • Sharing Rules cannot be specified to share data between User to User. To achieve this, the two users should be put in two groups and the data sharing rule should be specified to share data between these two groups.

8. Default Organization Field Access

Default Organization Field Access function is used to control the visibility of the fields in various modules for the entire organization. You can use this function to define your organization level business process to some extent from vtiger CRM user interface instead of programmatically defining the fields in various modules.

With default organization field access, one can either show or hide a field to the entire organization. Default Organization Field Privileges can be defined for the following modules:

Leads, Accounts, Contacts, Potentials, Activities, Trouble Tickets, FAQ, Price Books, Purchase Order,  Invoice, Notes, Emails, Products, Vendors, Quotes, and Sales Orders.

Notes

  • It is not possible to disable the mandatory fields in the modules. (for example, Last Name and Company fields in Leads module)
  • Default Organization field access overrides the profile level field access. For example your profile may allow to view the website field in leads. But you cannot view this field as it's access if denied in organization level field access
  • Default Organization level field access can be specified for custom fields also.
  • Changes made in Users, Roles, Profiles, Sharing Access and Field Access will not be reflected immediately. The user has to logout and login to  get the changes reflected.

9. Know Issues & Limitations

Security Management functions are not implemented for the following modules:

  • Calendar, Import, Export, Mass Delete, Change Owner, Change Status, Campaigns, Attachments and Outlook Plugin
  • Non-admin users should be allowed to view their Roles Subordinates, Profiles, and Groups.
  • When a group is deleted, handling to change the current ownership to the entities owned by this group should be done.
